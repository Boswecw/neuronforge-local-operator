# Schema Change Draft v1 — Lane Analytics Metric Provenance

## Revision

- Revision: 1
- Status: Draft
- Scope: `schemas/lane-analytics.schema.json`
- Change type: additive, non-breaking in intent

---

## Purpose

Define the first additive schema revision needed to make metric provenance machine-readable in NeuronForge lane records.

This draft exists to support the governance conclusion established in **Plan v2 — NeuronForge Lane Metric Semantics and Provenance**:

- the current fleet is operational
- metric semantics are not yet normalized across all lane types
- provenance must become explicit before wider lane expansion and downstream ingest

---

## Problem being solved

The current schema allows multiple lane types to publish common metric field names without declaring how those values were produced.

That creates risk of:

- false comparability
- governance opacity
- unsafe downstream aggregation
- inappropriate use of weak-evidence metrics in promotion logic

This schema change introduces explicit provenance metadata without forcing immediate redesign of the existing metric fields.

---

## Change summary

Add the following new fields to each lane record:

- `metric_provenance`
- `metrics_gate_eligible`
- `provenance_notes`

### Design intent

- `metric_provenance` declares the evidence class of the metric set
- `metrics_gate_eligible` declares whether the metric set may be used in automated or semi-automated promotion logic
- `provenance_notes` is required only for ambiguous or mixed evidence situations

---

## Proposed field definitions

## 1. `metric_provenance`

### Type

`string`

### Required

Yes

### Allowed values

- `instrument_derived`
- `benchmark_derived`
- `automated_heuristic`
- `operator_judged`
- `mixed`

### Meaning

#### `instrument_derived`

Metrics produced from structured instrumentation, deterministic scoring logic, or controlled evaluation harnesses tied to defined cases or runtime capture.

Use when values are generated by a governed system process rather than by human approximation.

#### `benchmark_derived`

Metrics produced from a static or versioned benchmark/test set with explicit scoring criteria.

Use when the primary evidence basis is an offline case pack, benchmark corpus, or frozen evaluation set.

#### `automated_heuristic`

Metrics produced through automated but non-authoritative scoring methods such as regex checks, rule-based grading, weak classifiers, or LLM-as-judge support that is not yet treated as benchmark-grade.

Use when the values are machine-produced but not strong enough to claim benchmark or instrument quality.

#### `operator_judged`

Metrics produced primarily by comparative manual review, operator scoring, or qualitative approximation.

Use when the metric values are visibility-supporting judgments rather than hard empirical measurements.

#### `mixed`

Metrics produced from a materially blended evidence basis where no single provenance class honestly describes the record.

Use only when multiple evidence classes substantially contribute to the recorded values.

---

## 2. `metrics_gate_eligible`

### Type

`boolean`

### Required

Yes

### Meaning

Declares whether the metric set is strong enough to participate in automated or semi-automated trust posture actions such as:

- auto-promote suggestions
- trust gate recommendations
- baseline replacement assist logic
- lane health gating

### Governance rule

This field is not a statement that promotion is automatic.

It is only a declaration that the metric evidence class is eligible to participate in governed promotion workflows.

### Default governance expectation

Expected default values by provenance class:

- `instrument_derived` → usually `true`
- `benchmark_derived` → usually `true`
- `automated_heuristic` → usually `false`
- `operator_judged` → `false`
- `mixed` → usually `false` unless explicitly justified

This is a governance expectation, not an absolute schema-level derivation rule.

---

## 3. `provenance_notes`

### Type

`string` or `null`

### Required

Yes

### Meaning

Human-readable explanation of provenance details that do not fit cleanly in the enum alone.

### Required usage cases

`provenance_notes` must be non-null when:

- `metric_provenance` = `mixed`
- a lane uses a novel evidence method that needs qualification
- `metrics_gate_eligible` is `true` for a provenance class that would normally default to `false`

### Optional usage cases

It may also be used to record:

- benchmark pack identity
- scoring caveats
- known weaknesses in the evaluation basis
- human review proportion in a mixed model

## Proposed schema constraints

## Required keys

The following new keys should be required:

- `metric_provenance`
- `metrics_gate_eligible`
- `provenance_notes`

`provenance_notes` remains nullable, but the key must exist.

This aligns with the existing nullability rule already used elsewhere in the lane record contract.

## Conditional validation rules

The validator should enforce the following governance conditions.

## Rule 1 — mixed requires notes

If:

- `metric_provenance` = `mixed`

Then:

- `provenance_notes` must be a non-empty string

## Rule 2 — operator judged cannot be gate eligible by default

If:

- `metric_provenance` = `operator_judged`
- `metrics_gate_eligible` = `true`

Then:

- validator should emit a hard failure unless an explicit policy exception exists

Preferred v1 stance:

- disallow this outright

## Rule 3 — automated heuristic true requires notes

If:

- `metric_provenance` = `automated_heuristic`
- `metrics_gate_eligible` = `true`

Then:

- `provenance_notes` must be non-empty
- validator should at minimum emit a warning or operator-attention message

Preferred conservative stance:

- allow only with explicit notes

## Rule 4 — mixed true requires notes

If:

- `metric_provenance` = `mixed`
- `metrics_gate_eligible` = `true`

Then:

- `provenance_notes` must be non-empty
- validator should emit at least a warning

Preferred conservative stance:

- default false unless the mixed evidence design is documented

---

## Governance interpretation rules

## 1. Provenance is about evidence class, not lane quality

A lane with `operator_judged` provenance is not necessarily low quality.

It simply has a different evidence basis and must not be treated as equivalent to benchmark- or instrument-backed metrics.

## 2. Gate eligibility is about automation safety, not trustworthiness alone

A lane may be approved for human-reviewed use while still being ineligible for automated promotion logic.

This is expected and healthy.

## 3. Shared numeric fields do not imply comparison safety

Even after this schema change, direct comparison still requires:

- compatible lane type
- aligned semantics
- aligned provenance
- materially comparable evaluation basis

This schema change improves visibility. It does not magically normalize all metrics.

## Recommended initial fleet mapping

## `continuity-progression-reasoning`

Recommended values:

- `metric_provenance`: `instrument_derived` or `benchmark_derived`
- `metrics_gate_eligible`: `true`
- `provenance_notes`: optional, but useful if the frozen case pack identity should be named

### Selection note

Use `benchmark_derived` if the primary basis is a frozen, versioned case pack.

Use `instrument_derived` if the current evaluation basis is better described as system-generated structured scoring tied to instrumented evaluation logic.

Do not pick both in v1.

Choose the dominant evidence class.

## `lore-safe-proofreading`

Recommended values:

- `metric_provenance`: `operator_judged`
- `metrics_gate_eligible`: `false`
- `provenance_notes`: non-null recommended

## `general-grammar-cleanup`

Recommended values:

- `metric_provenance`: `operator_judged`
- `metrics_gate_eligible`: `false`
- `provenance_notes`: non-null recommended

## Migration strategy

## Phase A — documentation first

- approve this schema change draft
- align governance docs with the new field meanings

## Phase B — additive schema update

- update `schemas/lane-analytics.schema.json`
- update `scripts/validate-lane-records.py`
- ensure validator error messages are explicit

## Phase C — lane record backfill

- update all existing lane records with the three new fields
- validate fleet-wide

## Phase D — downstream consumer discipline

- prevent ForgeCommand from aggregating incompatible metrics by default
- expose provenance visually in lane views and summaries

---

## Backward compatibility posture

This is intended as an additive revision.

However, once the schema is updated and the keys become required, existing lane records must be backfilled in the same change set to preserve fleet validity.

So the schema shape change is additive in design but should be operationally applied as a coordinated fleet update.

## Open decision points

## 1. Continuity lane provenance selection

Need final choice between:

- `instrument_derived`
- `benchmark_derived`

based on the dominant evaluation basis you want to encode in v1.

## 2. Hard failure vs warning for heuristic or mixed gate eligibility

Conservative recommendation:

- `operator_judged` + gate eligible = hard fail
- `automated_heuristic` + gate eligible = require notes and emit warning
- `mixed` + gate eligible = require notes and emit warning

## 3. Whether `provenance_notes` should be recommended or effectively mandatory for all non-benchmark / non-instrument lanes

Conservative recommendation:

- strongly encourage notes for all `operator_judged`, `automated_heuristic`, and `mixed` records
n- require notes only where ambiguity is highest in v1

---

## Compact decision

This schema change adds the minimum machine-readable governance structure needed to prevent metric provenance from remaining implicit.

It does not solve full lane-type metric normalization.

It does solve the immediate problem that a shared number without evidence context is unsafe for scaling, aggregation, and trust decisions.
