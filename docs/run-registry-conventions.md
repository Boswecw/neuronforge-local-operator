# Run Registry Conventions

## Purpose

This document defines the forward-use conventions for `registry/runs.md`.

The registry is the canonical success log for wrapper-driven proofreading runs.

It is not an attempt log.

---

## Scope

These conventions apply to new registry entries going forward.

Older entries may remain unchanged if they were valid at the time they were created.

---

## Success-only rule

`registry/runs.md` records successful runs only.

If `scripts/run-proofread.sh` fails, the wrapper exits before logging, so no success record should be written for that failed run.

---

## Record shape

Each run record should use this field order:

- `run id`
- `date`
- `model`
- `prompt file`
- `input file`
- `output file`
- `task`
- `notes`

Example shape:

    - run id: run-YYYY-MM-DD-NNN
      date: YYYY-MM-DD
      model: <model-name>
      prompt file: prompts/<prompt-file>.md
      input file: inputs/<input-file>.md
      output file: outputs/<output-file>.md
      task: <task-name>
      notes: <operator-notes>

---

## Spacing

Each new entry is appended with one blank line before it.

This matches the current behavior of `scripts/log-run.sh` and should be preserved for consistency.

---

## Field conventions

### `run id`

Use the format:

- `run-YYYY-MM-DD-NNN`

The numeric suffix should be zero-padded to three digits.

For wrapper-driven runs, prefer the value generated by `scripts/next-run-id.sh`.

### `date`

Use ISO date format:

- `YYYY-MM-DD`

For wrapper-driven runs, this is normally filled automatically from the system date.

### `model`

Record the literal model value used at execution time.

Example:

- `qwen2.5:14b`

### `prompt file`

Record the repository-relative path to the prompt file used for the run.

### `input file`

Record the repository-relative path to the input file used for the run.

### `output file`

Record the repository-relative path to the output file produced by the run.

For future wrapper-driven runs, prefer the wrapper-generated output naming pattern unless there is a deliberate reason to override it.

### `task`

Keep this short and functional.

Use `task` to describe what kind of run was being performed.

Examples:

- `proofread`
- `wrapper auto-output test`
- `wrapper auto-run-id test`

### `notes`

Use `notes` to capture why the run exists, what was being verified, or what was intentionally omitted or overridden.

Examples:

- `live run verification`
- `post-hardening live run`
- `run id and date omitted intentionally; wrapper should fill both`

---

## Output naming preference

For future wrapper-driven runs, prefer wrapper-generated output filenames of this form:

- `outputs/<sanitized-model>-<input-stem>-<run-id>.md`

This improves traceability between the output file and the registry entry.

Older manual output names may remain in the registry as part of the historical record.

---

## Historical preservation rule

Do not rewrite old registry entries just to force them into newer naming preferences.

The registry should preserve the actual historical record of what was run.

---

## Operator use rule

After a successful wrapper-driven run, verify that:

- the output file exists
- the output file is non-empty
- the expected run id appears in `registry/runs.md`
- the logged metadata matches the executed run
